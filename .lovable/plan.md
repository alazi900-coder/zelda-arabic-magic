

# خطة: إنشاء محلل بوكيمون متقدم (pokemon-text-parser.ts)

## المشكلة
المحلل الحالي `gfmsg-parser.ts` يستخدم خوارزمية XOR مبسطة ولا يدعم:
- اكتشاف الهيدر الديناميكي (3 استراتيجيات)
- فك ضغط zstd التلقائي
- كشف النصوص الفاسدة (Gibberish Detection)
- صيغة المتغيرات الصحيحة `[VAR XX_YYYY:params]`
- التعامل مع أسطر جديدة خاصة (0xBE00)

## الملفات المطلوبة

### 1. إنشاء `src/lib/pokemon-text-parser.ts` (~650 سطر)
محلل ثنائي كامل يتضمن:

**فك تشفير XOR المحدّث:**
- المفتاح الأساسي: `0x7C89`
- تقدم المفتاح لكل سطر: `+0x2983`
- تدوير داخل السطر: `key = (key << 3 | key >> 13) & 0xFFFF`

**اكتشاف الهيدر الديناميكي (3 استراتيجيات):**
- v2-u16: الأكثر شيوعاً — `u16 sections, u16 lines, u32 totalLen, u32 initKey=0, u32 sdo`
- v2-u32: للملفات الكبيرة — `u16 sections, pad, u32 lines, ...`  
- v1-flat: بسيط — `u32 lineCount` فقط

**فك ترميز النص:**
- `0x0000` → نهاية السطر
- `0x0010` → `[VAR XX_YYYY:params]`
- `0xBE00` / `0x000A` → سطر جديد `\n`
- باقي القيم → `String.fromCharCode`

**كشف النصوص الفاسدة:**
- إذا >30% من الأحرف هي أحرف تحكم أو Private Use → إعادة المحاولة بدون XOR

**التحقق من جدول الأسطر:**
- فحص 64 عينة للتأكد أن الإزاحات تشير لمواقع صالحة

**إعادة البناء (buildDat):**
- يكتشف صيغة الملف الأصلي
- يُشفّر النصوص الجديدة ويعيد بناء جدول الإزاحات

### 2. إنشاء `src/lib/zstd-utils.ts` (~50 سطر)
- فحص Magic bytes: `0x28 0xB5 0x2F 0xFD`
- فك ضغط تلقائي باستخدام `@bokuweb/zstd-wasm` (مثبّتة مسبقاً في المشروع)
- دالة `autoDecompressZstd(buffer)` تعيد `{ data, wasCompressed }`

### 3. تحديث `src/pages/PokemonProcess.tsx`
- استبدال استيراد `gfmsg-parser` بـ `pokemon-text-parser` و `zstd-utils`
- إضافة خطوة فك ضغط zstd تلقائية قبل التحليل
- دعم استراتيجيات القراءة المتعددة للتوافق مع Android
- عرض معلومات إضافية: نوع الهيدر المكتشف، هل كان مضغوطاً

### 4. الإبقاء على `gfmsg-parser.ts`
يبقى كملف منفصل لأنه يدعم صيغة MsgDataV2 الأصلية (GFMSG library format)، بينما `pokemon-text-parser.ts` يدعم الصيغة الفعلية المستخدمة في ملفات اللعبة المستخرجة.
